Captured 2026-07-10 from the P3 build + the skill-validation campaign, so it

## Org hardening (the headline effort)

The org pillar ships v0.5.0 **experimental** — the runtime is tested and the
dep-org acceptance case is real, but the discipline is a sketch, not
optimized. v0.6 is the evidence campaign that earns removing the label,
mirroring the authoring-skill methodology (author-from-skill-alone → judge →
fix → re-test):

- **Use cases** — 3–5 domains beyond investing/dependencies (incident/on-call
  per service, arXiv-topic tracking, competitor intel, inbox triage), each
  stood up by a fresh agent from the PUBLIC org-creation skill alone (the
  public skill has never had a stranger exit-test — v1.1's test was on the
  private predecessor).
- **Model variety** — org cycles across tiers (sol/terra/luna, sonnet-5, an
  open model) to find where brief quality / citation discipline degrades.
- **Judging criteria** (the hard part) — a measurable "good org" rubric:
  fact/judgment split, citation verifiability, falsifier concreteness, audit
  accuracy trend, null-wake honesty rate, lint-violation rate, severity
  calibration vs replayed ground truth, cross-model judge panels for brief
  usefulness. Use `org replay --pristine` as the controlled bed.
- Fold findings back into the skill + templates (the v0.5 cycle yielded 3
  template rules from ONE fresh-org cycle — expect a rich harvest).

## sync-skills for the codex orchestrator

Today `ultracodex sync-skills` writes only to `.claude/skills/`, which Claude
Code and opencode auto-discover; codex reads `AGENTS.md` and never sees it, so
a codex orchestrator has to be pointed at the skill file by hand (the tested
path) or wired through the codex plugin marketplace (unverified against our
repo). Easy fix: make `sync-skills` install per orchestrator.

- Prompt during setup (or a `--target` flag): install for **Claude &
  opencode** (`.claude/skills/`, today's behavior), **codex** (write/append a
  managed block to `AGENTS.md` that points at the bundled skill, or drop skill
  files where codex loads them), or **both**.
- Non-interactive default and a `--yes`/flag form for scripting.
- Keep the managed AGENTS.md block idempotent and clearly fenced so re-running
  is safe and the user's own AGENTS.md content is untouched.
- Verify the codex plugin-marketplace route end-to-end once the repo is public
  with current plugin metadata; if it works, offer it as a fourth install mode.
- Update docs/skills.md and the README skill pitch once codex is first-class.

## Residency as a per-seat policy

The resident-vs-fresh axis (ADR-0004 §11). `.thread` plumbing + Executor
Contract resume capability already ship; needs per-seat config
(residency: fresh|resident), periodic re-grounding, and replay semantics for
resident seats. Principle fixed: **workers may be resident; judges must be
fresh** (fresh gates caught real issues; resident builders get prefix-cache
reuse). Non-residency is what makes replay/audit work, so residency must be a
visible per-seat choice, never a default.

## opencode robustness (from the skill-validation campaign)

- **Per-backend concurrency caps** — opencode's server-per-call design
  collides on concurrent `serve` spawns (`serve exited before announcing a
  port` under load). The two skill-val driver deaths were this. Add
  `[backends.opencode] max_concurrency` and a spawn retry with backoff.
- **Live conformance on opencode 1.17.x** — the binary self-updated
  1.16.2 → 1.17.18 mid-campaign; the permission fix was validated live on
  1.17.18 but the 10-assertion conformance kit has not re-run there.
  TESTED_OPENCODE_VERSION stays 1.16.2 until it does. Re-run the live kit,
  then bump.
- **opencode data-dir override in the adapter** — the ~2 GB user db made
  every spawn heavy; isolate via a built-in XDG_DATA_HOME override option.
